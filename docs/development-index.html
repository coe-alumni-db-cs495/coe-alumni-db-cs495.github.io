Development Guide: How to Modify & Extend the Software

This guide is for developers who want to modify or extend the COE Alumni Database application. It covers the development environment setup, project structure, build processes, and contribution workflow.

Prerequisites

Before you begin, make sure you have:

  • .NET 8 SDK installed
  • Node.js 20.x installed
  • Git installed
  • A code editor (Visual Studio 2022 or Visual Studio Code recommended)
  • Basic knowledge of C#, React, and TypeScript

Setting Up Your Development Environment

Step 1: Clone the Repository

git clone [repository-url]
cd coe-alumni-db-cs495

Step 2: Install Backend Dependencies

cd coe-alumni-db-cs495.Server
dotnet restore

Step 3: Install Frontend Dependencies

cd ../coe-alumni-db-cs495.client
npm install

Step 4: Set Up the Database

The database file is gitignored, so you'll need to create it locally:

cd ../coe-alumni-db-cs495.Server
dotnet ef database update

This creates the SQLite database at ../coe-alumni-db-cs495.DB/AlumniDatabase.db with all necessary tables.

Database Location: The connection string in appsettings.json points to ../coe-alumni-db-cs495.DB/AlumniDatabase.db. Make sure the DB directory exists before running migrations.

Running the Application Locally

Option 1: Run from Visual Studio

  1. Open the solution file in Visual Studio 2022
  2. Set coe-alumni-db-cs495.Server as the startup project
  3. Press F5 or click Start Debugging
  4. The backend will start and automatically launch the React dev server

Option 2: Run from Visual Studio Code

  1. Open the project folder in VS Code
  2. Open two terminals
  3. Terminal 1 (Backend): 
    cd coe-alumni-db-cs495.Server
dotnet run
  4. Terminal 2 (Frontend): 
    cd coe-alumni-db-cs495.client
npm run dev

Option 3: Run Separately

You can also run backend and frontend in separate terminal windows:

# Terminal 1: Backend
cd coe-alumni-db-cs495.Server
dotnet run

# Terminal 2: Frontend
cd coe-alumni-db-cs495.client
npm run dev

Access Points

  • Frontend: https://localhost:52953
  • Backend API: https://localhost:5001 (or http://localhost:5000)
  • Swagger API Docs: https://localhost:5001/swagger

Build Management

Backend Build

The backend uses the .NET CLI and MSBuild:

# Restore dependencies
dotnet restore

# Build the project
dotnet build

# Build for production
dotnet publish -c Release -o ./publish

Frontend Build

The frontend uses npm scripts defined in package.json:

# Development mode with hot reload
npm run dev

# Production build
npm run build

Automated Builds

GitHub Actions automatically builds and deploys the application when code is pushed to the master branch. The workflow is defined in .github/workflows/build.yml.

The automated build process:

  1. Installs npm dependencies
  2. Builds the React frontend (npm run build)
  3. Copies frontend build to backend's wwwroot folder
  4. Restores .NET dependencies
  5. Publishes the .NET application
  6. Deploys to the server (on push to master)
Testing Builds: Always test your changes in build mode locally before pushing to master: 
cd coe-alumni-db-cs495.client
npm run build
The deployment runs in production mode, which may behave differently than development mode.

Database Management

Entity Framework Core Migrations

The project uses EF Core migrations to manage database schema changes.

# Create a new migration
dotnet ef migrations add MigrationName

# Apply migrations to database
dotnet ef database update

# Remove last migration (if not applied)
dotnet ef migrations remove

# View migration SQL
dotnet ef migrations script

Database Connection String

Connection string is configured in appsettings.json:

{
  "ConnectionStrings": {
    "DefaultConnection": "Data Source=../coe-alumni-db-cs495.DB/AlumniDatabase.db"
  }
}

Authentication & Security

The application uses JWT (JSON Web Tokens) for authentication:

  • JWT secret key is configured in appsettings.json
  • Passwords are hashed using BCrypt before storage
  • Tokens are sent in the Authorization header as Bearer tokens
  • Protected routes require valid JWT tokens
Security Note: The JWT key in appsettings.json is for development only. Production uses a different key stored securely. Never commit production secrets to the repository.

Making Changes

Adding a New Feature

  1. Create a new branch from master
  2. Implement your changes in both backend and frontend as needed
  3. Test locally in both dev and build modes
  4. Create a Pull Request to master
  5. The PR triggers a build-only workflow to check for errors
  6. After review and approval, merge to master
  7. Merging to master triggers automatic deployment

Adding a Backend Endpoint

  1. Create or modify a controller in Controllers/
  2. Add data models in Models/ if needed
  3. Update database context if database changes are needed
  4. Create and apply EF Core migration
  5. Test the endpoint using Swagger or Postman

Adding a Frontend Component

  1. Create component file in src/components/ or src/pages/
  2. Add TypeScript types in src/types/ if needed
  3. Connect to backend API via src/services/api.ts
  4. Add routing in App.tsx if it's a new page
  5. Test in development mode

Modifying the Database Schema

  1. Update your model classes in Models/
  2. Create a new migration: dotnet ef migrations add YourMigrationName
  3. Review the generated migration file in Migrations/
  4. Apply the migration: dotnet ef database update
  5. Test that your changes work correctly
  6. Commit both the model changes and migration files

Code Style & Standards

While there are no formal style guidelines currently enforced, follow these general practices:

C# Backend

  • Follow standard C# naming conventions (PascalCase for classes/methods, camelCase for parameters)
  • Use meaningful variable and method names
  • Keep controllers thin - move business logic to services
  • Use async/await for database operations
  • Add XML comments to public APIs

TypeScript/React Frontend

  • Use functional components with hooks
  • Define TypeScript interfaces for props and state
  • Keep components focused and single-responsibility
  • Use meaningful component and file names
  • Extract reusable logic into custom hooks

Testing

Automated testing is not currently implemented. When adding tests in the future, consider:

  • Backend: xUnit or NUnit for unit and integration tests
  • Frontend: Vitest (already configured in .esproj) or Jest for component tests
  • E2E: Playwright or Cypress for end-to-end testing
